# Definición de VSlices Tooling

## Decisión

VSlices Tooling vivirá como proyecto propio dentro de la suite VSlices.

Aunque nace desde una línea técnica de VSlices Docs Standard, no será tratado únicamente como una carpeta interna del estándar documental ni como una funcionalidad aislada de VSlices Framework.

VSlices Tooling evolucionará de la mano de VSlices Docs Standard y VSlices Framework, pero podrá crecer con identidad, ciclos de validación y artifacts propios.

## Contexto

Durante la iteración de tooling documental se observó una fricción concreta: mantener documentos, plantillas, niveles, scopes y convenciones editoriales de VSlices Docs Standard de forma manual puede generar duplicación, divergencia y pérdida de continuidad.

Inicialmente, esta fricción apareció dentro de VSlices Docs Standard. Sin embargo, al planificar el primer slice, se hizo evidente que la solución requiere tooling ejecutable, comandos, templates estructurados y posiblemente integración futura con otros productos de la suite.

Esto hizo necesario decidir si VSlices Tooling debía entenderse como una parte interna de Docs Standard, una capacidad de VSlices Framework o un proyecto propio.

## Problema o tensión

La tensión principal era evitar dos extremos:

* tratar VSlices Tooling como producto completo demasiado pronto
* esconderlo dentro de otro producto y limitar su evolución futura

Si se trataba como producto maduro desde el inicio, existía riesgo de sobreingeniería, diseño prematuro y pérdida de foco del primer slice.

Si se mantenía solo como una parte interna de Docs Standard o Framework, existía riesgo de mezclar responsabilidades: Docs Standard define lenguaje documental, Framework apoya implementación, y Tooling ejecuta tareas de soporte entre ambos.

## Razonamiento

Se escogió que VSlices Tooling viva como proyecto propio porque su responsabilidad no pertenece completamente a un solo producto existente.

VSlices Docs Standard define documentos, estructuras, niveles, convenciones, diagramas y caminos de continuidad.

VSlices Framework puede entregar infraestructura, CLI o integración técnica para proyectos .NET.

VSlices Tooling debe encargarse de convertir definiciones, templates o convenciones en operaciones ejecutables que ayuden a generar, conectar o mantener artifacts de VSlices.

Esta separación permite validar tooling desde slices pequeños sin forzar que cada experimento se convierta inmediatamente en parte oficial de Framework o Docs Standard.

## Tradeoffs

| Decisión o efecto                                                   | Ganancia                                                                   | Costo, riesgo o pérdida                                                    |
| ------------------------------------------------------------------- | -------------------------------------------------------------------------- | -------------------------------------------------------------------------- |
| VSlices Tooling vive como proyecto propio.                          | Puede evolucionar con identidad, ciclos de validación y artifacts propios. | Se agrega un proyecto más a mantener dentro de la suite.                   |
| VSlices Tooling nace desde una línea técnica de Docs Standard.      | Mantiene conexión directa con el dolor real que lo originó.                | Puede confundirse inicialmente con una parte interna de Docs Standard.     |
| VSlices Tooling evoluciona de la mano de Framework y Docs Standard. | Preserva continuidad entre documentación, metodología e implementación.    | Requiere cuidar límites para no mezclar responsabilidades entre productos. |
| El primer slice se mantiene pequeño.                                | Reduce sobreingeniería y permite validar valor rápido.                     | No resuelve todavía el tooling completo ni sus integraciones futuras.      |

## Consecuencias

VSlices Tooling tendrá documentación propia, decisiones propias y evolución propia.

El primer slice de VSlices Tooling se enfocará en generación documental desde templates estructurados, partiendo por un Documento de contexto L0 en español.

VSlices Docs Standard seguirá siendo responsable de definir qué documentos existen, cómo se estructuran, qué niveles tienen y qué convenciones editoriales deben respetar.

VSlices Tooling será responsable de ejecutar operaciones que usen esas definiciones, sin reemplazar el juicio documental humano.

VSlices Framework podrá integrarse más adelante con VSlices Tooling, pero no absorberá automáticamente todas sus responsabilidades.

## Condiciones de revisión

Esta decisión debe revisitarse si:

* VSlices Tooling no logra justificar su existencia como proyecto propio
* la implementación se vuelve más compleja de mantener que el valor que entrega
* VSlices Framework incorpora capacidades de tooling que vuelven redundante este proyecto
* VSlices Docs Standard cambia significativamente su modelo documental
* aparecen casos reales donde Tooling necesita operar más allá de documentación
* el primer slice no valida utilidad práctica
