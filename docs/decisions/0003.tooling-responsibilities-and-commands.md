# 0003. Separate tooling responsibilities and add operational commands

## Status

ProposeActive

## Context

VSlices Tooling started with a small deterministic slice focused on generating documentation artifacts from known templates.

The current direction includes commands such as:

```bash
vslices generate document --type context --level L0 --language es
vslices validate template --type context --kind document
vslices generate path --type software-project --language es
vslices evolve document --file /file.md
```

Future versions may introduce assisted review, continuity checks, and AI-assisted document generation.

However, these capabilities should not be mixed into the first deterministic commands.

VSlices Tooling must remain understandable, predictable, and safe to use before introducing assisted behavior.

## Decision

VSlices Tooling will separate the following responsibilities:

| Responsibility        | Purpose                                              |
| --------------------- | ---------------------------------------------------- |
| `generate`            | Create artifacts from known templates                |
| `validate`            | Verify deterministic rules                           |
| `evolve`              | Move an artifact to a deeper available level         |
| `review`              | Analyze quality, risks, and coherence                |
| `check continuity`    | Analyze relationships across documents               |
| `generate --assisted` | Guide document creation through assisted interaction |

The first iterations will focus on deterministic and operational commands.

AI-assisted behavior will be introduced only after generation, validation, inspection, and evolution are reliable enough.

## Initial operational improvements

VSlices Tooling should prioritize commands and flags that make the CLI easier to understand and safer to use.

Recommended additions:

```bash
vslices list templates
vslices list paths
vslices list levels --type context
vslices inspect document --file /file.md
vslices validate document --file /file.md
```

Recommended flags:

```bash
--output
--dry-run
--json
```

## Command semantics

### Generate

`generate` creates a new artifact from a known template.

It should not infer domain intent or modify existing documents.

Example:

```bash
vslices generate document --type context --level L0 --language es --output docs/context.example.md
```

### Validate

`validate` verifies deterministic rules.

It should produce expected errors and reproducible results.

Examples:

```bash
vslices validate template --type context --kind document
vslices validate document --file docs/context.example.md
```

### Evolve

`evolve` changes the structural level of an existing artifact.

It should preserve existing content and add the missing structure required by the next level.

It should not invent domain knowledge.

Examples:

```bash
vslices evolve document --file docs/context.example.md
vslices evolve document --file docs/context.example.md --to L1
```

### Review

`review` analyzes quality, coherence, risks, and possible misunderstandings.

It may be assisted by AI.

It should produce observations and suggestions, not deterministic validation errors.

Examples:

```bash
vslices review document --file docs/context.example.md --type context
vslices review path --file docs/domain-context.md --type domain-context
```

### Check continuity

`check continuity` analyzes relationships across documentation artifacts.

It may be assisted by AI.

It should focus on conceptual alignment between contexts, structures, use cases, decisions, vocabulary, and future evolution notes.

Example:

```bash
vslices check continuity --root ./docs
```

### Assisted generation

`generate --assisted` guides the user through document creation.

It may start from a minimal L0 skeleton, ask questions, organize answers into segments, and recommend moving to a deeper level when needed.

Example:

```bash
vslices generate document --type context --language es --assisted
```

This capability is intentionally deferred.

## Metadata requirements

Generated documents should include enough metadata to support later inspection, validation, evolution, and continuity checks.

Example:

```yaml
vslices:
  kind: document
  type: context
  level: L0
  language: es
  template: context.document
  templateVersion: 0.1.0
```

Templates should also be versioned.

Example:

```yaml
id: context.document
version: 0.1.0
type: context
kind: document
```

## Consequences

This decision keeps the first tooling iterations small and deterministic.

It also creates a clear path for future assisted behavior without mixing generation, validation, review, and continuity analysis into a single command.

The main tradeoff is that some useful assisted workflows will arrive later.

This is acceptable because VSlices Tooling should first become reliable, inspectable, and understandable before becoming intelligent.

## Follow-up

Future decisions should define:

* the canonical CLI command grammar
* the difference between document templates and path templates
* the structure of generated artifact metadata
* the expected output format for validation errors
* the first supported `review` reports
* the first supported `check continuity` scope
